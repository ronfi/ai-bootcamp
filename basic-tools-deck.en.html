01

Terminal: Learn to Talk to Your Computer

Graphical interfaces are for "users"; the terminal is for "commanders". In a GUI you can only do what the developers pre-built; in the terminal you can issue any instruction — and AI speaks the language natively.

01 · Terminal

Terminal = a text chat box; Shell = the interpreter that gets what you mean

Terminal

A window where you type instead of click. You enter a command, it returns a result — that's the whole deal.

Shell (command interpreter)

The program doing the actual work inside the terminal: Linux usually runs bash, macOS defaults to zsh, Windows has PowerShell. Different dialects, same core grammar.

Why it's worth learning

① Precise — one command is one unambiguous instruction; ② Copy-pasteable — send it to someone and it just runs; ③ Servers have no GUI; ④ it is the native language of AI agents.

$ echo "Hello, terminal"
Hello, terminal
$ ls
training  stock-report  notes.txt
$ # anything after the hash is a comment explaining the command
$ ▌
Pep talk: the terminal won't break because you typed something wrong — most mistakes just print "command not found". Type boldly.
01 · Terminal · How to open it on each platform

Three platforms, three entrances — Windows folks, just install WSL

macOS built-in

⌘ + Space, search "Terminal", hit Enter. Default shell is zsh. Power users can add iTerm2.

Linux built-in

On the desktop press Ctrl+Alt+T; on a cloud server, ssh in and you're already inside the terminal itself.

Windows use WSL

The built-in PowerShell works (comparison table on slide 13), but we strongly recommend WSL — a real Linux running inside Windows.

# Open PowerShell as Administrator, one command:
PS> wsl --install
Installing: Ubuntu ...
# After a reboot, "Ubuntu" appears in the Start menu
# Open it = you now own a full Linux terminal

Why this step pays off

With WSL installed, every command in the rest of today (tmux / git / Claude Code / Hermes) can be copied verbatim; over 95% of the world's servers run Linux — learning it is learning the lingua franca.

WSL = Windows Subsystem for Linux, an official Microsoft feature, available on Windows 10 1809+ / Windows 11

01 · Terminal · Look at the map before you walk

The file system is a tree — and you're always standing on some node

Absolute path: start from the root

/home/tom/training — the full address starting at /, valid from anywhere.

Relative path: start from where you stand

Standing in ~, it's just training; .. = one level up, . = right here.

Tilde ~ = home directory

Your private turf (/home/yourusername). Lost? cd ~ takes you home.

01 · Terminal · Command set 1: find your way

pwd where am I · ls what's here · cd go there

$ pwd                # print working directory — where am I
/home/tom
$ ls                 # list what's here
training  stock-report  notes.txt
$ ls -l              # -l for details (size/time/permissions)
$ ls -a              # -a to include hidden files
$ cd training        # change directory — enter a folder
$ pwd
/home/tom/training
$ cd ..              # go up one level
$ cd ~               # one hop home

The lost-person combo

Whenever you don't know where you are, what's around, or where to go — pwd → ls → cd. Three moves and you're oriented.

Tab completion = half the typing

Type cd tr, press Tab, it completes to training/. Nobody types full paths by hand.

↑ arrow = history

Press ↑ to recall the last command, tweak two characters, hit Enter.

01 · Terminal · Command set 2: reading files

cat dump it all · less page through · tail -f keep watching

$ cat notes.txt      # print the whole file (great for short ones)
Today's to-dos: install WSL, learn tmux...
$ more notes.txt     # page by page: Space for next, q to quit
$ less big.log       # more, upgraded: ↑↓ scroll, / search, q to quit
$ head -5 big.log    # just the first 5 lines
$ tail -5 big.log    # just the last 5 lines
$ tail -f agent.log  # follow new lines live (Ctrl+C to stop)

Which one?

Short file: cat; long file: less (remember q to quit); only care about the ends: head / tail.

tail -f is the agent-watching tool

Claude Code and Hermes both write logs while grinding through long tasks — tail -f is their live commentary feed.

01 · Terminal · Command set 3: make, move, delete

mkdir make · cp copy · mv move · rm delete (careful)

$ mkdir demo             # make directory — new folder
$ mkdir -p a/b/c         # -p builds all three levels at once
$ touch todo.txt         # create an empty file
$ cp todo.txt backup.txt # copy
$ cp -r demo demo2       # copying a folder needs -r
$ mv todo.txt done.txt   # move — rename and relocate in one
$ mv done.txt demo/      # move into a folder
$ rm demo/done.txt       # remove a file
$ rm -r demo2            # deleting a folder needs -r

⚠️ rm has no recycle bin tattoo it on your DNA

Deleted means gone. ls the target before deleting; never, ever run an internet-copied rm -rf /-style command — have AI double-check it first.

Habit: back up important folders first

Before surgery, cp -r project project.bak — five seconds buys you an undo. Chapter 4's git is the ultimate version of this idea.

01 · Terminal · Command set 4: vi survival guide

vi: you don't need to master it, just survive it

The scenario: you ssh into a server to change one config line, and vi is often all there is. It has an "insert" mode and a "command" mode — every ounce of beginner panic comes from not knowing which one you're in.

① Open

vi filename — you start in command mode; typing doing nothing is normal

② Start editing

Press i (insert); -- INSERT -- appears at the bottom; now you can type

③ Stop editing

Press Esc to return to command mode (a few extra presses never hurt)

④ Save / flee

:wq save and quit; :q! quit without saving — the escape hatch

$ vi config.txt
# press i → make your edits → press Esc → type :wq and Enter
$ # done, and you made it out alive

Can't memorize it? Normal.

Save these four survival moves to your phone and you're covered. Leave everyday editing to AI (next chapter's Claude Code edits files far faster than you); vi is for emergencies only.

01 · Terminal · Command set 5: ssh remote login

ssh: turn a server a thousand miles away into this very window

$ ssh tom@203.0.113.7      # username@server-address
Are you sure ...? yes        # type yes on first connect
tom@203.0.113.7's password: # type the password (nothing showing is normal)
Welcome to Ubuntu 24.04 LTS
tom@server:~$              # the prompt changed = you're on the server

# —— Passwordless login (configure once, enjoy forever) ——
$ ssh-keygen                  # generate a key pair (just keep hitting Enter)
$ ssh-copy-id tom@203.0.113.7  # deliver your public key to the server

# —— Transferring files ——
$ scp localfile tom@203.0.113.7:~/   # upload
$ scp tom@203.0.113.7:~/remotefile .  # download

Read the prompt to know where you are

When the prompt turns from local$ into tom@server:~$, every command you type runs on the server; exit brings you back home.

Your phone can ssh too

Install an app like Termius and you can reach your server from the subway — this is the key to "commanding agents remotely" later on.

Planting a hook 🪝

The moment ssh drops, the program running on the remote end dies with it. What now? Next chapter.

01 · Terminal · ssh, level 2: keys and the address book

The public key is the lock, the private key is the key, config is the address book

$ ssh-keygen -t ed25519       # generate a key pair (passphrase optional)
$ ls ~/.ssh
id_ed25519  id_ed25519.pub  config  known_hosts
#  ↑private key = the key, never share   ↑public key = the lock, share freely
$ cat ~/.ssh/id_ed25519.pub   # the public key looks like:
ssh-ed25519 AAAAC3Nza... tom@laptop
# what ssh-copy-id does = append this line to the server's
#   ~/.ssh/authorized_keys (the "locks that open this door" list)

$ cat ~/.ssh/config           # SSH's address book:
Host work
    HostName 203.0.113.7
    User tom
    IdentityFile ~/.ssh/id_ed25519
$ ssh work                    # four letters onto the server, forever after
$ ssh-add                     # passphrase on the key? the keychain remembers it

Division of labor between the pair

Public key (.pub) = the lock: installed on the server's authorized_keys door; private key = the key: stays on your machine to open it. Handing out the public key is harmless; a private key leaving your computer is like posting your house key to a group chat.

~/.ssh/config = the address book

Give every server a nickname; write down address, username and which key just once — after that ssh work, scp file work:~/ all use it. A must-have once you manage two or more servers.

ssh-add = the keychain

A passphrase makes the private key safer — but typing it on every connect? ssh-add hands the key to ssh-agent for safekeeping, so you type it once per boot; ssh-add -l shows what's hanging on the ring.

Reading authorized_keys in reverse: adding one public-key line = cutting someone a copy of your house key. So on slide 33, when Claude Code helps edit configs, any change to this file deserves your personal approval.
01 · Terminal · Command set 6: three ways to call for help

Forgetting commands isn't embarrassing — knowing how to ask is the real skill

Move 1: --help

Almost every command accepts it: ls --help, tmux --help — one screen of usage at a glance.

Move 2: man

man ls opens the full manual; q quits. Classic, but long-winded.

Move 3: ask AI the 2026 way

Paste the full error message to AI; or just state the goal: "convert every mp4 in this folder to mp3" — let it write the command, you make sure you understand it before hitting Enter.

$ claude -p "count the txt files in the current directory with one command"
ls *.txt | wc -l
The new division of labor: you describe goals + review commands; AI memorizes the syntax. The one red line: for deletions or system-level commands you don't understand, make AI explain every flag before running.

02

tmux: Close the Laptop, the Work Keeps Running

The hook planted last chapter: when ssh drops, the remote program dies. tmux keeps your session living on the server — a disconnect just means "not watching right now", not "start all over".

02 · tmux · Why and what

A "safe-deposit box" that keeps your session on the server

The tragedy without tmux

You ssh in and set an agent on a 3-hour task → step into an elevator, Wi-Fi drops for 3 seconds → the task dies on the spot; start from scratch.

Daily life with tmux

The task runs inside a tmux session → disconnect, shut the laptop, go home and sleep, none of it matters → next day tmux attach, everything exactly as you left it.

Three layers

Session > Window (like browser tabs) > Pane (one screen split into pieces). Beginners: just remember Session.

02 · tmux · The four survival moves

Four moves is all: new, detach, list, attach

$ sudo apt install tmux   # install (Mac: brew install tmux)
$ tmux new -s ai          # ① start a session named ai
# ...launch your long task inside...
# ② detach: press Ctrl+b, release, then d (detach)
[detached (from session ai)]
$ tmux ls                 # ③ list all sessions
ai: 1 windows (created Fri Jul  4 10:02)
$ tmux attach -t ai       # ④ reattach — scene restored, untouched

# Bonus: split panes
# Ctrl+b %  split left/right | Ctrl+b "  split top/bottom
# Ctrl+b arrow keys  move between panes

The command key Ctrl+b

Every tmux shortcut is "press Ctrl+b, release, then one key". Today you only need d (detach).

Detach ≠ close

After detach the program keeps running; typing exit inside the session is what actually closes it.

Name your sessions

Name them by purpose: tmux new -s claude, -s hermes — recognizable at a glance when you come back.

02 · tmux · The point of this chapter

tmux is the agent's dorm: once it moves in, it never clocks out

Daytime · office

ssh to the server, tmux new -s cc, set Claude Code refactoring the project.

Evening · on the subway

Ctrl+b d to detach, close the laptop. The task keeps running on the server. Want progress on your phone? Termius ssh in, tmux attach.

Late night · at home

Open your home computer, tmux attach -t cc — the afternoon's scene is back in one second, and the agent is already done.

One line: ssh gives you the power to show up; tmux gives you the freedom to walk away. Chapter 5's official Remote Control takes this experience one level higher.
03

Scheduled Jobs: Put Your Computer on an Alarm Clock

tmux solves "you leave, the program stays." cron solves "you're away, the program starts itself." Master it and your server gets a daily routine.

03 · Scheduled jobs · crontab syntax

crontab: five numbers, one timetable

$ crontab -e            # edit my job table (picks an editor first time)
# ┌─ min(0-59) ┌─ hour(0-23) ┌─ day ┌─ month ┌─ weekday(0-7)
#    │           │            │      │        │
  0    9    *    *    *   ~/backup.sh >> ~/logs/backup.log 2>&1
# ↑ run the backup at 09:00 daily, log output and errors
*/10   *    *    *    *   ~/fetch-data.sh
# ↑ fetch data every 10 minutes
  0   22    *    *    5   ~/weekly-report.sh
# ↑ generate the weekly report every Friday at 22:00
@reboot  tmux new -d -s hermes 'hermes gateway start'
# ↑ bring the butler back up whenever the server reboots
$ crontab -l            # list scheduled jobs

The five-field chant

minute · hour · day · month · weekday. * = every, */10 = every 10; commas list, dashes span. Can't remember? Ask AI: "write me a cron line for Mondays at 8 a.m."

Logging is not optional

End every line with >> logfile 2>&1 — otherwise a job that fails at 3 a.m. fails silently forever. Chapter 2's tail -f is exactly what watches it.

Two rookie traps

Cloud servers usually run on UTC (8 hours behind Beijing time); cron ships with almost no environment variables, so scripts should use absolute paths.

03 · Scheduled jobs · Where it shines

Scheduled jobs × AI: three ways to run the night shift

① A data pipeline on a timer

Fetch market data after every close, clean it, store it — Exercise 1 from Lecture 1 in its final form; then trigger the analysis script once the data lands.

② Summon Claude on schedule

Headless mode goes straight into cron: claude -p "read last night's error log, summarize in 3 bullets" — AI works on a schedule with nobody in the room.

③ Hermes' calendar

hermes cron puts your resident butler on a schedule: morning briefings, backups, watching webpages for changes — cron with memory and skills on top.

The formula: cron decides "when," the script or agent decides "what," and tmux + logs guarantee "you can find out what happened." There's a homework check-in for this one.

Git / GitHub: a Time Machine for Code

rm has no recycle bin, but git does — the most powerful one in the universe. It's also the collaboration protocol of programmers worldwide, and the best-looking résumé you'll ever own.

04 · Git / GitHub · Why everyone should know it

Four reasons, each more down-to-earth than the last

① Time machine the undo button

Every commit is a save point. Broke something? One command returns you to any save. The feature you wish essays had.

② Collaboration protocol worldwide

Tens of thousands of people co-writing one project without chaos — that's git's branching and merging. The "common tongue" of programmers everywhere.

③ The AI-era safety net today's focus

You dare let Claude Code slash boldly through your code because of git: every AI change is recorded, and one click rolls it back. Without git, even AI won't work with a free hand.

④ GitHub = the new résumé most practical for students

Employers increasingly check GitHub before the résumé. Push the exercises and projects you've built — that's a verifiable portfolio.

Concepts in one line: git is the version-control tool installed on your machine; GitHub is a website hosting repos in the cloud (Gitee, GitLab are its peers).
04 · Git / GitHub · Mental model

Three steps: edit → save → sync to the cloud

add = drop changes into the "to-save" basket

commit = click! — a save point, with a one-line note

push / pull = sync with the cloud: push up / pull down

04 · Git / GitHub · The daily five

A day in the life of a repo

$ git init                 # turn this folder into a git repo (once)
$ git status               # the catchphrase: what's going on right now?
modified: index.html
$ git add .                # stage all the changes
$ git commit -m "finish signup form"  # save + one line of plain words
$ git log --oneline        # view the save history
a3f9c21 finish signup form
9b2d1e0 initial version
$ git diff                 # what exactly changed this time?
$ git checkout a3f9c21 -- index.html  # the undo: restore an old version

The catchphrase: git status

Whenever you're unsure, run status first — it tells you where you are, what changed, and what you can do next.

Commit messages say "why"

"changed index.html" ❌ (git already knows); "fix untappable button on mobile" ✅ (future you says thanks).

How small is one commit?

Finish one describable small goal, save once. Small steps, frequent saves — that's what makes the time machine useful.

04 · Git / GitHub · Three things in the cloud

GitHub: backup, showcase, collaboration — one-stop shop

# ① Clone someone else's project (say, the "Lobster")
$ git clone https://github.com/openclaw/openclaw.git
Cloning into 'openclaw'...

# ② Push your own repo to the cloud (create an empty repo on GitHub first)
$ git remote add origin https://github.com/you/my-site.git
$ git push -u origin main   # first push; afterwards just git push

# ③ Continue on another computer / sync teammates' changes
$ git pull

Sign up and it's yours

A free github.com account = unlimited public repos. Pick your username seriously — it will appear on your résumé.

README.md is the storefront

The repo homepage renders the README automatically: what the project is, how to run it, screenshots. A repo only counts as a portfolio piece once the README is good.

Star = bookmark + like

Last session's 380k stars for the Lobster and 210k for Hermes — this is that star. Starring the good projects you use is also how you socialize in this world.

Bonus egg 🥚: GitHub's official CLI gh — everything the website does, done in the terminal. gh auth login once; then gh repo create my-site --public --source . --push handles "create repo + link + push" in one command, and gh pr create opens PRs. Unlock it after the three moves on the left feel natural.

05 · Claude Code · Safety & in-session permission requests

The permission prompt isn't a nuisance — it's the intern raising a hand

> clean up the old logs I don't need
Claude wants to run:

    rm -rf ~/logs/2025/

  [y] Allow once   [a] Always allow this session   [n] Deny
# Stop — read the command before you pick. rm -rf
# deserves three extra seconds: right folder? collateral damage?

When it asks

Editing files, running commands, reaching the network — anything "for real" requests approval first; read-only work (browsing code, searching) never interrupts you.

Three rules for approving

① Understand it before you approve it — especially rm, git push, curl | bash; ② money, deletion, anything public-facing: always approve manually, one by one; ③ safe read-only commands that keep recurring can be allowlisted via /permissions to cut the noise.

⚠️ Match the mode to the setting

"Auto-accept edits" is plenty for daily coding; --dangerously-skip-permissions (skip everything) belongs only inside a container or VM — the flag's name is telling you not to.

One-line principle: treat permissions like package deliveries — read the slip before you sign. The confidence to let AI loose comes from one gate held firmly: dangerous actions always pass through your hands.
